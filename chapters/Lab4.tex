\section{Assignment}
After covering RTOSs and their components in detail in the lecture and learning how to schedule periodic tasks on QNX, this assignment focuses on getting started developing for QNX on the Raspberry Pi 4.
After getting familiar with the hardware setup and remote debugging, the goal is to create 2 periodic tasks using the POSIX API and monitor their execution using the system profiler and an external logic analyzer.
Code for this implementation as well as those of the following tasks can be found in the GitHub repository for this course \footnote{\url{https://github.com/feldbacherSimone/Assignment_04}}.

\section{Approach}
Since the assignment is focused on working with threads rather than the precise code inside of them, simplified or dummy algorithms were used for some of the tasks. 
\texttt{clock\_nanosleep} was used to achieve the periodic execution of the tasks.
\begin{CCode} 
struct timespec next; 
clock_gettime(CLOCK_MONOTONIC, &next);
for(;;)
{
	// Sample Accelerometer Data

	// Advance deadline by period

	next.tv_nsec += 1000000; // 1 ms
	if(next.tv_nsec >= 1000000000)
	{
		next.tv_sec += 1;
		next.tv_nsec -= 1000000000;
	}
	clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next, NULL);
}
\end{CCode}

\subsection{In class task - measurement with logic analyzer}
During the lecture a simple task was set up to toggle one of the GPIO pins on the Raspberry Pi. Using a logic analyzer the timing of the toggling was measured and compared to the expected timing. The results are shown in Figure~\ref{fig:lab4-logic-analyzer}. Surprising was that even when using a tiny delay (few nanoseconds) the period would increase to almost 2ms. This was caused by the minimum timer resolution on the QNX image being 1ms. Any delay is rounded up to the next multiple of 1ms, which is why the period increased to almost 2ms.

\subsection{Task 2: CRC16 Checksum} 
For the checksum task, a CRC16-CCITT implementation was written to compute a checksum over a simulated 1MB flash segment, represented as a static array filled with a dummy byte pattern \texttt{0xA5}. The polynomial \texttt{0x1021} with an initial value of \texttt{0xFFFF} was used. The resulting 2-byte checksum is then encrypted using AES-128 in CTR mode via the tiny-AES library before being sent over the serial line. The task runs periodically every 500ms using \texttt{clock\_nanosleep()}.
The execution time of the checksum task was measured using the system profiler, 
as shown in Figure~\ref{fig:profiling-t2}. The dominant cost is the CRC16 
computation over the 1MB flash segment, with a WCET of approximately 60ms and a worst case response 
time of 70ms, well within the 500ms period.

\subsection{Task 3: Diagnostic Task}
The diagnostic task receives and decrypts incoming commands using AES-128 CTR

\subsection{Shared Resources and Priority Inversion}
The only shared resource in the system is the serial line, accessed by both the 
checksum task (Task 2) and the diagnostic task (Task 3). Access is protected 
using a \texttt{pthread\_mutex}. A bounded priority inversion can occur when 
Task 2 holds the mutex and is preempted by Task 3, which then blocks waiting 
for the mutex to be released. Since Task 2 holds the mutex only briefly, the 
inversion duration is negligible. The AES key and IV are read-only constants 
and require no protection.

A deadlock is impossible in this system as there is only one shared mutex. 
Deadlock requires a circular wait between at least two resources, which cannot 
form with a single mutex.

\subsection{Alternative Resource Protection Methods}
Alternative methods for protecting shared resources include:
\begin{itemize}
    \item \textbf{Priority ceiling protocol} -- raises the priority of the mutex 
    holder to the ceiling priority, preventing priority inversion entirely.
    \item \textbf{Spinlocks} -- busy-wait instead of blocking. Wastes CPU cycles 
    and is unsuitable for an RTOS where other tasks need the processor.
    \item \textbf{Disabling interrupts} -- prevents all preemption while the 
    resource is held. Too aggressive for an RTOS as it blocks all other tasks 
    including higher priority ones.
    \item \textbf{Lock-free data structures} -- avoid locking entirely using 
    atomic operations. Complex to implement correctly and overkill for a simple 
    2-byte serial write.
\end{itemize}
A plain mutex was chosen as it is simple, well understood, and sufficient given 
the short duration the resource is held and the large timing margins available.
