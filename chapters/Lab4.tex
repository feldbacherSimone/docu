\section{Assignment}
After covering RTOS and their components in detail in the lecture and learning how to schedule periodic tasks on QNX, this assignment focuses on getting started developing for QNX on the Raspberry Pi 4.
After getting familiar with the hardware setup and remote debugging, the goal is to create 2 periodic tasks using the POSIX API and monitor their execution using the system profiler and an external logic analyzer.
code for this implementation as well as those of the following tasks can be found in the GitHub repository for this course \footnote{\url{https://github.com/feldbacherSimone/Assignment_04}}.

\section{Approach}
Since the assignment is focused on working with threads rather than the precise code inside of them, simplified or dummy algorithms were used for some of the tasks. 
\texttt{clock\_nanosleep} was used to achieve the periodic execution of the tasks.
\begin{CCode} 
struct timespec next; 
clock_gettime(CLOCK_MONOTONIC, &next);
for(;;)
{
	// Sample Accelerometer Data

	// Advance deadline by period

	next.tv_nsec += 1000000; // 1 ms
	if(next.tv_nsec >= 1000000000)
	{
		next.tv_sec += 1;
		next.tv_nsec -= 1000000000;
	}
	clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next, NULL);
}
\end{CCode}

\subsection{In class task - measurement with logic analyzer}
During the lecture a simple task was set up to toggle one of the GPIO pins on the Raspberry Pi. Using a logic analyzer the timing of the toggling was measured and compared to the expected timing. The results are shown in Figure~\ref{fig:lab4-logic-analyzer}. Surprising was that even when using a tiny delay (few nanoseconds) the period would increase to almost 2ms. This was caused by the minimum timer resolution on the QNX image being 1ms. Any delay is rounded up to the next multiple of 1ms, which is why the period increased to almost 2ms.
